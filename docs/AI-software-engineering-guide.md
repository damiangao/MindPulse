# AI 辅助软件工程指南

> 基于 claude-chat 项目实践的经验总结

## 背景

我们在开发 claude-chat 项目时遇到了典型问题：
- "看起来简单"的功能实际上很复杂
- 多 worktree 并行导致合并冲突
- Bug 反复出现，修了又坏
- 过于依赖 AI，自己没深入参与
- 缺少测试覆盖，上线后才发现问题

这份文档的目的是建立一套可重复的工作流，避免这些问题。

---

## 核心原则

### 1. Spec 不是装饰品

**问题**：写完 spec 就忘了，直接让 AI 开干；或者认为"简单功能不需要细看 spec"。

**正确姿势**：
- Spec 是你和 AI 之间的**合同**
- 复杂功能要先分解，分解到每个子任务都能清楚定义验收标准
- 即使是"业界有参考"的功能，也要明确指出：
  - 我们的特殊需求是什么
  - 哪些部分可以直接参考，哪些需要调整
  - **已知风险**和**需要验证的假设**

### 2. 小步增量，不追求完美

**问题**：追求一次性实现完整功能，导致改动太大难以 review 和测试。

**正确姿势**：
- 每个 worktree 只做一件事
- 分阶段交付：先跑起来，再优化
- 每次合并前确保核心测试通过

### 3. 自己要深入参与

**问题**：完全交给 AI，自己没理解代码怎么工作的。

**正确姿势**：
- 不是让你写所有代码，而是让你**理解每一个决策**
- AI 给的建议，你要问"Why"
- 关键路径（认证、支付、数据隔离）必须 human-in-the-loop

### 4. 测试先行

**问题**：只有后端测试，前端测试刚刚加上；E2E 测试是出问题后才引入的。

**正确姿势**：
- 测试是功能的"Executable Spec"
- 每加一个功能，先想清楚"怎么验证它工作"
- 单元测试 + 集成测试 + E2E 测试 各有用途

### 5. 谨慎使用 Worktree

**问题**：多个 worktree 并行导致合并时出大问题。

**正确姿势**：
- Worktree 适合"隔离实验"，不适合"并行开发完整功能"
- 尽量在一个 worktree 里做，用 branch 管理不同功能
- 合并前在本地跑完整测试套件

#### Worktree 使用场景判断

| 场景 | 推荐方式 | 原因 |
|------|---------|------|
| 开发新功能 | 单 worktree + branch | 完整上下文，review 简单 |
| Bug 修复 | 单 worktree + branch | 改动小，不需要隔离 |
| 技术方案验证 | Worktree（临时） | 需要对比多个方案 |
| 危险的重构 | Worktree（临时） | 隔离风险，不影响主分支 |
| 并行开发多个功能 | 禁止 | 合并冲突是主要损失来源 |

#### 正确流程

**创建实验 worktree：**
```bash
# 从 master 创建实验分支
git worktree add .claude/worktrees/feature-x feature-x

# 实验完成后删除 worktree
git worktree remove .claude/worktrees/feature-x
git worktree prune  # 清理孤立的 worktree 引用

# 如果实验成功，合并回主分支
git checkout master
git merge feature-x
git branch -d feature-x
```

**合并前检查清单：**
- [ ] 在 worktree 里跑完整测试套件
- [ ] 没有未提交的修改
- [ ] 与 master 没有冲突（或冲突已手动解决）
- [ ] 功能已完成并通过自测

#### 常见问题与应对

**冲突太多无法合并：**
1. 回到 master，基于 master 创建新 branch
2. 逐 commit cherry-pick 有用的改动
3. 放弃过于复杂的 worktree 分支

**Worktree 忘记清理：**
- 定期运行 `git worktree list` 检查孤立 worktree
- 删除前确认没有未合并的改动
- 用完立即 remove，不要留着"以后可能用"

**多个 worktree 造成困惑：**
- 养成习惯：每次只在一个 worktree 里工作
- 切换前确认上一个 worktree 没有未完成的工作
- 用有意义的名字：`feature-auth`, `experiment-retry-logic`

#### claude-chat 项目教训

**错误做法（已踩坑）：**
- 开了多个 worktree 同时开发多个功能
- 各 worktree 修改了相同的文件（session.py, app.jsx 等）
- 合并时冲突几十个，手动解决后引入了新 bug

**正确做法：**
- 所有功能在一个 worktree 里，用 branch 区分
- 每次只活跃一个 branch
- 合并前确保测试通过

## 开发流程

### 功能开发流程

```
1. 需求阶段
   ├── 明确功能边界（做什么 / 不做什么）
   ├── 识别已知风险和需要验证的假设
   └── 分解为可独立交付的子任务

2. 设计阶段
   ├── 写 Spec（包含架构、数据流、接口）
   ├── human review spec（你自己要读）
   └── 识别与现有代码的边界

3. 实现阶段
   ├── TDD（先写测试，再实现）
   ├── 小步提交（每个 commit 可独立理解）
   └── 本地跑测试验证

4. 验收阶段
   ├── 你做人工测试
   ├── 检查是否破坏了现有功能
   └── 合并前跑完整测试套件
```

### Bug 修复流程

```
1. 复现阶段
   ├── 明确复现步骤
   ├── 识别触发条件
   └── 确认预期行为 vs 实际行为

2. 调查阶段
   ├── 理解相关代码（不要只看 bug 周围）
   ├── 追溯根本原因（不是只修症状）
   └── 提出 2-3 个可能的修复方案

3. 修复阶段
   ├── 选择最小改动方案
   ├── 写测试防止 regression
   └── 验证修复有效

4. 确认阶段
   ├── 跑完整测试套件
   ├── 确认没有引入新问题
   └── 写清楚 commit message
```

#### Bug 调查 Checklist

**复现阶段：**
- [ ] 能稳定复现 bug（在独立环境中验证）
- [ ] 记录了复现步骤：什么操作 → 什么结果 → 预期 vs 实际
- [ ] 识别了触发条件：什么情况下会触发
- [ ] 明确了影响的用户场景

**调查阶段：**
- [ ] 阅读了相关代码，不只是 bug 周围的代码
- [ ] 追溯到根本原因，不是表面症状
- [ ] 列出了 2-3 个可能的修复方案
- [ ] 评估了每个方案的复杂度和对系统的影响
- [ ] 记录了假设和需要验证的点

**修复阶段：**
- [ ] 选择了最小改动方案
- [ ] 先写了能复现 bug 的测试（如果测试缺失）
- [ ] 修复后验证测试通过
- [ ] 手动验证 bug 已修复

**确认阶段：**
- [ ] 跑完整测试套件（单元 + 集成 + E2E）
- [ ] 确认没有引入新问题
- [ ] 写清楚 commit message：描述根因，不是表面修复
- [ ] 更新了相关的文档或注释

#### 常见 Bug 类型与应对

| 类型 | 症状 | 根因 | 应对 |
|------|------|------|------|
| 竞态条件 | 偶发，测试不稳定 | 并发访问共享状态 | 加锁或用不可变数据结构 |
| 边界情况 | 特定输入触发 | 没处理空值/极限值 | 补全边界测试 |
| 集成问题 | 单独测试通过，整体失败 | 模块间接口不匹配 | 补集成测试 |
| 回归问题 | 修一个坏一个 | 测试覆盖不足 | 补 regression 测试 |
| 架构问题 | 改了这里，那里坏了 | 耦合太高 | 重构边界，明确职责 |

## AI 辅助的正确方式

### 让 AI 做什么

| 任务 | AI 适合度 | 说明 |
|------|-----------|------|
| 代码生成 | 高 | 有明确 spec 时效率很高 |
| 代码审查 | 高 | 发现潜在问题和风格问题 |
| 测试生成 | 高 | 可以快速生成覆盖 |
| 技术调研 | 高 | 搜文档、找参考 |
| Bug 调查 | 中 | 需要你提供上下文 |

### 让人类做什么

| 任务 | 人类必须参与 | 说明 |
|------|-------------|------|
| 架构决策 | 是 | 影响长期维护成本 |
| 安全相关 | 是 | 认证、授权、数据隔离 |
| 业务逻辑 | 是 | 确保符合业务需求 |
| Code Review | 是 | 作为最后一道防线 |

### 与 AI 协作的技巧

**不要这样问**：
- "帮我修这个 bug"
- "实现这个功能"

**应该这样问**：
- "这个 bug 的根因是 X，我尝试了 Y 但不行，请分析还有哪些可能"
- "这个功能 Spec 在第 3 节定义了 Z，你的实现是否符合，请对照检查"
- "我担心这部分代码会在并发时出问题，请帮我分析"

## 测试策略

### 分层测试

```
E2E 测试 (Playwright)
├── 覆盖核心用户流程
├── 不追求 100% 覆盖
└── 关键路径必须通过

集成测试 (pytest)
├── 覆盖 API、WebSocket、数据层
├── 每个接口至少一个测试
└── 测试真实交互（不 mock 数据库）

单元测试 (pytest + Vitest)
├── 覆盖独立函数、组件
├── 快速执行
└── 关注边界情况
```

### 测试何时写

- 新功能：**先写测试再实现**（TDD）
- Bug 修复：**先写能复现 bug 的测试，再修**
- 重构：**确保测试覆盖再动**

## 常见错误与应对

### "看起来简单"

**错误**：看到业界有参考，就认为不需要详细设计。

**应对**：
- 即使是参考实现，也要明确我们的差异点
- 在 Spec 里写"已知风险"

### "Worktree 并行"

**错误**：开多个 worktree 同时开发功能，最后合并冲突一堆。

**应对**：
- 优先在一个 worktree + branch 模式
- Worktree 只用于隔离实验（如尝试不同的技术方案）
- 合并前确保在本地跑完整测试

### "反复修同一个 bug"

**错误**：修完上线，没加 regression 测试，又坏了。

**应对**：
- 每个 bug 修复都要有对应的测试
- 跑完整测试套件再合并

### "测试是后面再加的"

**错误**：先写功能代码，后来才补测试。

**应对**：
- 改变思维：测试是功能的一部分
- 新功能必须同时有测试，否则不算完成

## Skill 使用规范

> claude-chat 项目有很多 skill，但之前没有用起来，导致走了很多弯路。

### 为什么要用 Skill

Skill 是经过验证的最佳实践封装。使用 skill 意味着：
- 不需要从头思考"这个功能该怎么做"
- 遵循已验证的工作流，避免常见错误
- 在适当的时机触发适当的工具

### Skill 触发时机

| 场景 | 应该用的 Skill | 说明 |
|------|---------------|------|
| 想实现一个功能，不知道怎么做 | **brainstorming** | 先设计，再实现 |
| 需要写详细的实现计划 | **planner** | 复杂功能需要多步骤计划 |
| 开发新功能或修复 Bug | **tdd-guide** | 先写测试，再实现 |
| 写完了代码，需要 review | **code-reviewer** | 质量检查 |
| 代码涉及安全（认证、支付、数据） | **security-reviewer** | 必须过安全关 |
| Build 失败了 | **build-error-resolver** | 快速定位问题 |
| 需要运行 E2E 测试 | **e2e-runner** | 关键用户流程验证 |
| 要写新功能文档 | **doc-updater** | 保持文档同步 |

### Skill 使用检查清单

在每个开发任务中：

**开始前：**
- [ ] 确认是否有合适的 skill 可以用
- [ ] 调用 skill 而不是凭感觉做事

**过程中：**
- [ ] 按照 skill 定义的流程走
- [ ] 使用 skill 提供的 checklist

**结束后：**
- [ ] 验证是否遵循了 skill 的指导
- [ ] 如果有偏差，记录原因

### /simplify Skill 使用指南

**触发条件：** 代码写完、测试通过、准备 commit 前

**正确流程：**
1. 代码写完，测试通过
2. 运行 `/simplify`（自动修复质量/效率问题）
3. **必须重新跑测试** — 验证修复没有引入新 bug
4. Commit

**注意：** `/simplify` 会自动改代码，改了就可能引入问题。跑测试是强制要求，不能跳过。

```
实现代码 → 跑测试通过 → /simplify → 重新跑测试 → commit
```

**为什么不能跳过测试：**
- 自动修复可能改变逻辑
- 自动修复可能引入新 bug
- 没有测试验证的修改是危险的

### brainstorming Skill 使用指南

**触发条件：** 想实现一个功能，无论是"简单"还是"复杂"

**正确流程：**
1. 调用 brainstorming skill
2. 探索项目上下文
3. 问问题（一次一个）
4. 提出 2-3 个方案
5. 呈现设计方案
6. 写 design doc 并 commit
7. 用户 review 通过后 → 调用 writing-plans skill

**不要跳过：**
- "这个功能太简单了，不需要设计" — 简单功能最容易踩坑
- "我大概知道怎么做，直接让 AI 实现吧" — 没有设计就实现容易返工

### tdd-guide Skill 使用指南

**触发条件：** 开发新功能、修复 bug、重构

**正确流程：**
1. 调用 tdd-guide skill
2. 写测试（RED）
3. 运行测试，确认失败
4. 写实现（GREEN）
5. 运行测试，确认通过
6. 重构（IMPROVE）
7. 验证覆盖率 80%+

**不要跳过：**
- "先实现功能，后面再补测试" — 测试是功能的一部分
- "功能太简单，不需要 TDD" — 小功能更需要测试防止 regression

### code-reviewer Skill 使用指南

**触发条件：** 写完代码后，commit 前

**正确流程：**
1. 调用 code-reviewer skill
2. 等待 review 结果
3. 修复 CRITICAL 和 HIGH 问题
4. 确认没有 BLOCK 问题后可以合并

**必须 review 的情况：**
- 安全相关代码（认证、授权、支付、数据）
- 架构变更
- 合并到 shared branch 前
